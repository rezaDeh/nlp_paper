Abstractify: a NLP Approach to Creating Abstracts for Scientic Papers
Julian Beaulieu Junru Liu Mason Jerome Yuncheng Yu Drake Johnson Reza Dehkordi
Abstract
In this paper, we present multiple summariza-
tion algorithms of the extractive and abstrac-
tive kind. Abstractive models focus on cre-
ating a summary of the input text from the
ground-up. Whereas, extractive models focus
on pulling out the most important parts of text
found within the document to best represent
it as a summary. These methods each have
their strengths and weaknesses. Finding a bal-
ance can be tricky to nd when choosing which
model to use for summarization. That is why
our group, Abstractify, developed an easy-to-
use Web App that allows for quick input of
ones paper, and it allows for the selection of
the model to use for comparison between gen-
erated abstracts.
1 Introduction
Mason Jerome
Text Summarization is an ongoing area of research
in Natural Language Processing. There are two
main subsets of text summarization models: ab-
stractive and extractive. Abstractive models fo-
cus on building up a generative model that creates
a summary of the input text from the ground-up.
Whereas, extractive models focus on pulling out
the most important parts of text found within the
document to best represent it as a summary.
These methods each have their strengths and
weaknesses. Extractive models often generate bet-
ter n-gram similarity than their abstractive counter-
parts. However, human readability can suffer as a
result. Finding a balance can be tricky to nd when
choosing which model to use for summarization.
That is why our group, Abstractify, developed an
easy-to-use Web App that allows for quick input of
ones paper, and it allows for the selection of the
model to use for comparison between generated
abstracts.
2 Problem Background
Mason Jerome
While text summarization is already a well-
developed task, we wanted to utilize it for a novel
method. For this, we looked into creating a text
summarization for academic papers that can de-
velop a corresponding abstract. While this may not
be needed by everyone writing a paper, it can pro-
vide an outside perspective on what ones abstract
could look like, through which some inspiration
may be derived.
With the dataset, we worked on developing a sim-
ple, easy-to-use framework through which anyone
could quickly put in their paper to get an abstract
in return. This required setting up a Web app to
copy the papers body into that connected to our
models for abstract results.
3 General Challenges
Julian Beaulieu, Mason Jerome, Yuncheng Yu
The rst challenge of developing a framework
through which we can produce summarized ab-
stracts was gathering a proper dataset to train and
test on. After checking through what was avail-
able online and considering creating our own from
scratch, we decided to use the arXiv dataset avail-
able through the TensorFlow dataset library. This
dataset contains over 200,000 supervised training
samples with the papers text body and the corre-
sponding abstract for its label. Through the training
process, these can be validated with 6,000+ data
samples held within the validation set. Finally, the
hold-out test set contains another 6,000+ samples
which can be used for evaluation. This large of
a dataset provides enough samples for all of our
models to train on to get a good generalized text
summarizer for our project.
The second big challenge we encountered was
during the training of the abstractive models, as the
size of the input and output of the models hinged on
the size of Google Collabs GPU sizes. If the input,
1
output, or both sizes were too big, the model would
get too large to train on the GPU. Thus, we had
to limit the size of the input and output to roughly
1000 characters for the input and roughly 100-125
characters for the output. Thus, requiring us to
chunk up the data to feed in the entire paper.

4.2 User Interface
Yuncheng Yu, Julian Beaulieu
The project is using a micro-framework, Flask,
which controls both the frontend and backend. The
frontend is written in regular JavaScript, CSS, and
HTML, while the backend is constructed using
python. When initially opening the website, the
user will land on the home page, which allows for
creating an abstract for the paper. In the top bar,
the user sees two other menu items which lets them
navigate to the Team and to the Model Overview
pages. When creating the abstract, the user has cur-
rently two ways of uploading their unnished paper
to Abstractify. The rst is by copy and pasting the
papers content into the input eld, the second be-
ing that they can simply drag and drop a PDF le
into the input eld, which will then get parsed and
converted to plain text before being rendered in
the input eld. The user then has the ability to
perform any last minute edits, such as removing
any artifacts which were not correctly parsed by
the PDF parser. From here, the user can choose
which abstractication method they would like to
use to create their abstract. From here, when the
user presses the Summarize button, a get request
is sent to an API endpoint from the ask server
which creates the abstract and then returns it to
the frontend. The frontend renders the returned
abstract in the abstract eld, allowing the user to
edit the abstract before copying it to their paper.
The user also has the option to clear the abstract
eld or both the input and abstract eld.
4.3 Framework Architecture
Yuncheng Yu, Julian Beaulieu
The project is using a micro-framework, Flask,
with both backend and frontend design imple-
mented. The users can enter the URL to load and
access the webpage. After uploading or pasting the
contents of a le to the input section, the webpage
will send a GET request to the ask server which
includes the choice of model to use, and the content
of the paper. After the ask server receives the re-
quest, it will determine which choice the user made
concerning the model to create the abstract with.
From there, it passes the data along to a function
dedicated to the chosen model. This is because
different models have different requirements, such
as needing the input to be chunked up. Depending
on if the model needs to have the input size reduced
or not, the input rst gets chunked up, then each
chunk is fed through the model before concatenat-
ing the output back to one abstract. In the event
that the extractive model is used, the entire input
is passed into the model at once, and the resulting
abstract is then output as a whole. The ask server
then receives the abstract from the model and re-
turns it using the get request. The JavaScript will
then render the abstract in the correct eld.

4.5 Extractive
Drake Johnson
The extractive model that we employed is an unsu-
pervised model that uses cosine similarity to rank
the sentences of a paper and create the summary
based on the highest ranking of those sentences.
The rst step of this approach is to separate the
paper into individual sentences. Once the sen-
tences have been separated, we build a similarity
matrix connecting each sentence to each other. Af-
ter weve built the similarity matrix, we go through
all combinations of two sentences to calculate their
similarity. When calculating the similarity, we rst
create a non-zero vector for each sentence that is
the size of all words in both combined sentences.
Once the vectors are made, we add one to each in-
dex of the sentence vector where the word appears.
This step also accounts for stop words by skipping
them when iterating through each sentence. Once
both vectors have been nalized, we calculate the
cosine distance between the two vectors. This score
is then placed into our similarity matrix, acting as
an edge weight between the two sentences. After
all cosine distances have been calculated, we se-
lect the top ve ranking sentences to generate our
summary. We chose to use the top ve sentences
because when testing, the ROUGE scores began to
decline as more sentences were added. To nd the
top ranked sentences, we used the HITS algorithm.
HITS is a link analysis algorithm commonly used
to rank search results, similar to the PageRank al-
gorithm. Both HITS and PageRank were tested in
the implementation, however there was no notice-
able difference in the resulting summaries. After
performing HITS on the similarity matrix, we end
up with our top ve sentences which make up our
summary.
4.6 Abstractive T5
Mason Jerome
The T5 model is the Text-to-Text Transfer Trans-
former model that utilizes a simple encoder-
decoder transformer for modular, multi-use lan-
guage modeling. I wanted to create a model via
transfer learning for the summarization task for cre-
ating abstracts and decided to use this model due
to its high learning-representation transferability.
Searching for a trained model, I found that Hug-
ging Face had a trained model on a variety of tasks
such as sentiment analysis, sentence completion,
question answering, as well as a few other Natural
Language Processing tasks. Which task it performs
is set off by a prompt that the T5 model trains on.
For summarization, it is tasked with summarize:
prior to every input.
With the model being built for high transferabil-
ity, transfer learning to learn a summarization task
suits it well. It just needs a good dataset to train
on. With that in mind, I utilized the arxiv dataset
to learn how to summarize papers into abstracts.
One of the struggles that needed to be overcome
is the input limitations that comes with transformer-
based models. After tokenizing the data and setting
up the ROUGE metrics, the training had to be done
on chunks of the articles and abstracts rather than
the entire papers all at once. This ensured that the
model could split the paper into parts, perform the
summarization task on each individual part, and
combine the predicted summaries rather than just
performing summarization on a truncated paper.

4.7 Abstractive BART
Junru Liu
BART (Lewis et al., 2019) is a pretrained model
with a bidirectional encoder and an autoregressive
decoder based on Transformer. Although works
satisfactorily for comprehension tasks, it demon-
strates exemplary performance in particular after
ne-tuning on text generation, such as abstractive
text summarization. Various versions of BART
ne-tuned on different datasets for different tasks
has been released, such as bart.base, bart.large, and
bart.large.mnli (Ott et al., 2019). The pretrained
model selected for this project is bart.large.xsum,
which is BART model with 12 encoder and decoder
layers ne-tuned on Xsum (Narayan et al., 2018).
Xsum is a dataset consists of high abstractive sum-
maries of news. In spite of this, this model is still
used since we would like to apply transfer learn-
ing considered the similarity between news and
academic paper.
Due to the limited computing power and fea-
tures of abstract, the maximum length of the input
and output were set as 128 and 1024. Moreover,
the original arXiv (Cohan et al., 2018) was also
sampled, from 203037, 6436, 6440 documents to
10000, 300, and 300 documents in the training, val-
idation, and test sets respectively. In addition, the
number of epochs was also limited to only three to
reduce training time.
4.8 Abstractive PEGASUS
Reza Dehkordi
Well known transformer models are capable of pro-
ducing language learning when combined with self-
supervised pre-training, specially when ne-tuned
on a variety of language tasks. Previously, the self
supervised nature of these models were established
in a general manner without any consideration for
the down stream tasks. with the down stream tasks
in mind, scientists at Google introduce PEGASUS,
a Transformer encoder-decoder model with a pre-
training self-supervised objective with the purpose
of improving ne-tuning performance on abstrac-
tive summarization.
4
In the self-supervised objective, entire sentences
are removed from the documents and the objective
of the model is to recover these to the best of its
ability, which is a demanding task even for humans.
However, this challenge "encourages the model to
learn about language and general facts about the
world, as well as how to distill information taken
from throughout a document in order to generate
output that closely resembles the ne-tuning sum-
marization task" (Zhang et al., 2019). Utilizing this
form of self-supervision allows for the creation of
the same number of examples as the amount of doc-
uments lacking human annotation. This inability
has been a bottleneck in pure supervised models.
Masking "important" sentences resulted in the
best accuracy in the supervised samples. These
sentences are identied by nding ones that are
most similar to the rest of the text, using ROUGE
(Lin, 2004) as a metric for computing sentence
similarity.
With these adjustments to self-supervised ob-
jective. Training and testing PEGASUS on well
known summarization datasets resulted in state-of-
the-art performance while using 5% the number
of parameters as some of the benchmarks such as
T5. For our purpose we extracted the pre-trained
PEGASUS model from Hugging Face, trained and
tested its performance on the arXiv data also avail-
able through the application.

5.1 Original Abstract
This paper presents a simple unsupervised learn-
ing algorithm for classifying reviews as rec-
ommended (thumbs up) or not recommended
(thumbs down). The classication of a review
is predicted by the average semantic orientation
of the phrases in the review that contain adjec-
tives or adverbs. A phrase has a positive semantic
orientation when it has good associations (e.g.,
subtle nuances) and a negative semantic orien-
tation when it has bad associations (e.g., very
cavalier). In this paper, the semantic orientation
of a phrase is calculated as the mutual information
between the given phrase and the word excellent
minus the mutual information between the given
phrase and the word poor. A review is classied
as recommended if the average semantic orien-
tation of its phrases is positive. The algorithm
achieves an average accuracy of 74% when evalu-
ated on 410 reviews from Epinions, sampled from
four different domains (reviews of automobiles,
banks, movies, and travel destinations). The accu-
racy ranges from 84% for automobile reviews to
66% for movie reviews.

7 Conclusion
Yuncheng Yu
The text summarization research has started since
1958 by Lun using a statistical approach called
word frequency diagrams, and soon until now it
becomes one of the major research topic many
researchers are willing to contribute (Widyassari
et al., 2022). We have implemented two types of
text summarization models, abstractive and extrac-
tive, in our paper. For the abstractive models, we
6
used encoder-decoder pretrained models, such as
T5, BART, and PEGASUS. For the implemented
extractive model, it is an unsupervised method,
which uses the cosine similarity to measure the dis-
tance between sentences and ranked with a HIST
algorithms. All the models are evaluated with
ROUGE metrics, specically are ROUGE-L and
ROUGE-1. We used the ROUGE-L to measure
the longest matching sequence of words with the
longest common subsequence and ROUGE-1 to
measure the overlap of each word between the refer-
ence summaries and system. From our results, the
PEGASUS is the most ideal model, since the both
the length of content of the abstract is similar to the
original abstract. The other abstractive models and
extractive model also did their job, but abstractive
approach performs better since it contains more
relevant contents from the original context and the
generated abstract sounds more natural. Further-
more, We also deployed our model with Flask web
framework to allow users to either input their con-
text or upload their document in the entry box and
by selecting different models to output a copyable
abstract for users as a reference. For the future im-
provement, the team plans to explore more abstrac-
tive models, evaluate models with other state-of-art
elevation metrics, and enhance our user interface.
